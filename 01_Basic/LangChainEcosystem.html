01

What is LangChain & Why It Exists

The raw OpenAI API can generate text. But building a real application ‚Äî one that reads your documents, remembers conversations, uses tools, and handles multi-step reasoning ‚Äî requires a lot of glue code. LangChain is that glue.

Core Idea: LangChain is a framework for building applications powered by language models. It standardizes the way you connect LLMs to data sources, tools, memory, and other LLMs ‚Äî so you don't reinvent the wheel every time.

üß© The Problem LangChain Solves

Without LangChain, building a chatbot over your own documents means manually writing: document loading, text chunking, embedding creation, vector storage, retrieval logic, prompt construction, conversation memory management, output parsing, and error handling. That's hundreds of lines of boilerplate for something that should be ~20 lines.

üîë The Key Mental Model

Think of LangChain as a set of standardized interfaces and composable building blocks. Every piece (model, retriever, tool, memory) implements the same Runnable interface, so they can be snapped together like Lego. This is called LCEL ‚Äî LangChain Expression Language.

üì¶ What LangChain Gives You

Integrations with 100+ LLMs and vector stores ¬∑ Prompt management ¬∑ Memory ¬∑ RAG pipelines ¬∑ Agents and tools ¬∑ Streaming ¬∑ Async support ¬∑ Observability (LangSmith) ¬∑ Deployment (LangServe) ¬∑ Multi-agent orchestration (LangGraph)

‚ö†Ô∏è Important caveat: LangChain moves fast. The codebase has been refactored significantly (v0.1 ‚Üí v0.2 ‚Üí v0.3). Always check you're reading docs for the version you're using. As of 2025, LCEL is the canonical way to build chains. Older patterns using LLMChain are legacy.
02

Package Structure ‚Äî What to Actually Install

LangChain is split into several focused packages. This was done intentionally to keep your dependency footprint small and to allow the ecosystem to evolve independently.

ALWAYS langchain-core

The foundation. Contains all base abstractions: Runnable, BaseMessage, PromptTemplate, BaseRetriever. Never use directly ‚Äî it's pulled in automatically.

ALWAYS langchain

The main package. Contains higher-level chains, agents, memory, and document transformers. This is what you import day-to-day.

OPTIONAL langchain-openai

Official OpenAI integration. Provides ChatOpenAI and OpenAIEmbeddings. Use this instead of langchain-community for OpenAI.

OPTIONAL langchain-community

300+ third-party integrations: loaders, vector stores, tools, LLMs. Large package ‚Äî install only what you need via extras.

OPTIONAL langgraph

Multi-agent workflow orchestration using graphs. Use this for any stateful, cyclical, or multi-agent workflow.

OPTIONAL langsmith

Observability SDK. Tracing, evaluation, and debugging. Pairs with the LangSmith cloud platform.

Install for 90% of projects: 
# Core stack ‚Äî start here
pip install langchain langchain-openai

# Add for RAG projects
pip install langchain-community faiss-cpu

# Add for agents + complex workflows
pip install langgraph

# Add for observability
pip install langsmith
Why separate packages? Before the split, pip install langchain would pull in 100+ dependencies including heavy ML libraries even if you only needed basic chat. Now you only install what you actually use.

LangChain vs LlamaIndex vs Raw API Calls

This is one of the most common questions. The answer isn't "X is always better" ‚Äî it's about what you're building.

Dimension Raw OpenAI API LangChain LlamaIndex
Best for Simple, focused tasks Agents, workflows, diverse use cases Data-heavy RAG over large document corpora
Learning curve ‚úì Minimal ~ Moderate ~ Moderate
RAG tooling ‚úó None ‚úì Good ‚úì Excellent
Agent/tool use ~ Manual ‚úì Excellent ~ Basic
Multi-agent flows ‚úó None ‚úì LangGraph ~ Limited
LLM flexibility ‚úó OpenAI only ‚úì 100+ LLMs ‚úì Many LLMs
Observability ‚úó DIY ‚úì LangSmith ~ Basic tracing
Abstraction level Low (full control) Medium-high Medium-high

Decision Flow ‚Äî What Should You Use?

Simple one-off LLM call? (summarize, classify, translate)
‚Üí
Raw API
Building a chatbot, agent, or multi-step workflow?
‚Üí
LangChain
Primary goal: RAG over massive document collections?
‚Üí
LlamaIndex
Complex multi-agent state machines with cycles?
‚Üí
LangGraph
Need fine-grained control + already know LLMs well?
‚Üí

üí° Real talk: LangChain and LlamaIndex are not mutually exclusive. Many production systems use LlamaIndex for sophisticated document indexing and retrieval, then pipe results into a LangChain agent. Use both if needed.
04

Setting Up Your Environment & API Keys

Do this right from day one. Sloppy environment setup causes bugs that waste hours.

1

Create a virtual environment

Never install LangChain globally. Use a dedicated environment per project.

2

Install packages

Install the minimal set you need. Don't pip install langchain-community unless you specifically need it.

3

Set up your .env file

Store all API keys here. Never hardcode them. Never commit this file to Git.

4

Enable LangSmith tracing (optional but recommended)

Set LangSmith env vars from day one. You'll thank yourself when debugging.

# Step 1: Create virtual environment
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate

# Step 2: Install packages
pip install langchain langchain-openai python-dotenv
# .env file ‚Äî never commit this!
OPENAI_API_KEY=sk-...

# Optional: LangSmith observability
LANGCHAIN_TRACING_V2=true
LANGCHAIN_API_KEY=ls__...
LANGCHAIN_PROJECT=my-project
# Add .env to .gitignore immediately
echo ".env" >> .gitignore
# In your Python code
from dotenv import load_dotenv
import os

load_dotenv()  # loads .env into environment

# LangChain reads OPENAI_API_KEY automatically
# but you can also access it directly:
api_key = os.getenv("OPENAI_API_KEY")

# Quick sanity check ‚Äî your first LangChain call
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-4o-mini")
response = llm.invoke("Say hello in 5 words.")
print(response.content)
‚ö†Ô∏è Common mistakes: Using gpt-4 when you mean gpt-4o (much cheaper/faster). Forgetting load_dotenv() before instantiating LangChain objects. Setting OPENAI_API_KEY after the import statements (too late ‚Äî set env vars before importing).

üîë Which API Keys You'll Need

OpenAI ‚Äî OPENAI_API_KEY ‚Äî platform.openai.com
Anthropic ‚Äî ANTHROPIC_API_KEY ‚Äî console.anthropic.com
LangSmith ‚Äî LANGCHAIN_API_KEY ‚Äî smith.langchain.com
Pinecone ‚Äî PINECONE_API_KEY ‚Äî only if using Pinecone vector DB
Cohere ‚Äî COHERE_API_KEY ‚Äî only if using Cohere reranking

‚úÖ Verify your setup works by running the 6-line sanity check above. If you get a response back, you're ready to build.

Key Takeaways

The 5 things to internalize from this section

1.

LangChain is glue, not magic. It standardizes the interface between LLMs, data, tools, and memory. Understanding this prevents over-engineering.

2.

Install only what you need. Start with langchain + langchain-openai. Add packages as your project grows.

3.

Use LCEL from day one. Ignore old tutorials using LLMChain. The prompt | model | parser pattern is the modern standard.

4.

Raw API is fine for simple tasks. Don't reach for LangChain when a single openai.chat.completions.create() call would do.

5.

Enable LangSmith early. Two env vars and you get full observability ‚Äî invaluable for debugging complex chains and agents.
